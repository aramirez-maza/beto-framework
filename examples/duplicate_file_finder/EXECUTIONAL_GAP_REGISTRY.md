## 1. PURPOSE

Este registro documenta todos los BETO_GAP_EXECUTIONAL activos detectados
durante la ejecución del EXECUTION_READINESS_CHECK sobre las OQs críticas
del sistema Duplicate File Finder CLI.

Un BETO_GAP_EXECUTIONAL ocurre cuando una OQ crítica no alcanza el estado
PASS_EXECUTABLE ni PASS_WITH_LIMITS después de aplicar el
EXECUTION_READINESS_CHECK con los 8 campos obligatorios, y después de agotar
el ciclo de repreguntas tipadas (max_operational_requestions = 2).

Este registro es un artefacto complementario del CIERRE_ASISTIDO_OPERATIVO.md.
No reemplaza al BETO_CORE. No modifica el BETO_SYSTEM_GRAPH.

Compatibilidad BETO_PARALELO: cada gap es LOCAL a la unidad PARALLEL
propietaria de la OQ. Un gap activo en una unidad no bloquea globalmente
otras unidades cuyas OQs no tienen gaps activos.

---

## 2. GAP STATUS LEGEND

```
ACTIVE         Gap detectado, sin resolución declarada. Bloquea materialización
               de la unidad propietaria.

RESOLVED       Gap cerrado mediante respuesta del operador que alcanzó
               PASS_EXECUTABLE o PASS_WITH_LIMITS en re-evaluación.

ESCALATED      Gap transferido al operador para decisión fuera del alcance
               del cierre asistido. Materialización suspendida hasta declaración.
```

## 6. RESOLUTION PROTOCOL

Para cerrar un gap activo, el operador debe:

```
Paso 1: Declarar explícitamente la respuesta a la OQ correspondiente.
        La respuesta debe ser específica y no contener patrones blandos
        no evaluables (ej. "alto", "adecuado", "cuando sea necesario").

Paso 2: El asistente ejecuta re-evaluación del EXECUTION_READINESS_CHECK
        con los 8 campos sobre la OQ resuelta.

Paso 3: Si el resultado es PASS_EXECUTABLE o PASS_WITH_LIMITS:
          - El gap se cierra con status RESOLVED
          - La OQ se actualiza a DECLARED_EXECUTABLE o DECLARED_WITH_LIMITS
          - El BETO_CORE se actualiza en Sección 8 o Sección 9 según aplique
          - El Trace ID pendiente se genera como ID autorizado

Paso 4: Si el resultado sigue siendo FAIL_EXECUTIONAL_GAP:
          - El gap permanece ACTIVE
          - Se registra la respuesta parcial como intento fallido
          - Se documenta por qué la respuesta no alcanzó nivel ejecutable

Paso 5: Una vez cerrados todos los gaps de una unidad:
          - La unidad actualiza su Gate G-2B a APPROVED_EXECUTABLE
            o APPROVED_WITH_LIMITS
          - La materialización (Paso 8) queda desbloqueada para esa unidad
```

**Prioridad de resolución recomendada:**

```
1. GAP-003 (OQ-5 — lenguaje) — impacto global. Resolver primero.
2. GAP-002 (OQ-4 — symlinks) — impacta schema de output de SCANNER
   que es input de HASHER.
3. GAP-005 (OQ-8 — permisos) — resolver antes de materializar SCANNER
   y HASHER.
4. GAP-001 (OQ-1 — algoritmo hash) — resolver antes de materializar
   HASHER.
5. GAP-004 (OQ-6 — formato reporte) — resolver antes de materializar
   REPORT_COMPOSER. Gate G-2B OSC obligatorio.
```
